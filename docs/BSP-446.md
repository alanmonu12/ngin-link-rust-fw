# 📘 BSP-F446 (Board Support Package) - Documentación Interna

## 🎯 Objetivo Principal
El crate `bsp-f446` actúa como la **Capa de Abstracción de Hardware (HAL ajustada)** específica para nuestra placa basada en el microcontrolador **STM32F446**.

Su propósito principal es inicializar el microcontrolador (relojes, pines, interrupciones y periféricos base) y devolver "Drivers" limpios a la aplicación principal (`main.rs`). De esta forma, la lógica del protocolo (USB y CAN) no necesita saber nada de registros, velocidades de reloj o qué pines físicos se están usando.

---

## 🏗️ Estructura del Crate
El crate está compuesto por tres archivos principales:

### 1. `lib.rs` (El Orquestador)
Expone la estructura principal `Board` y la función pública `init()`.
- **Configuración de Relojes (RCC y PLL):** Configura el microcontrolador para usar un cristal externo (HSE) de 8 MHz en modo bypass (común en placas Nucleo). A través del PLL (Phase-Locked Loop), multiplica esta frecuencia para alimentar el sistema central (Sysclk) a 84 MHz y genera un reloj exacto de 48 MHz, el cual es un requisito estricto para que el periférico USB funcione correctamente.
- **Inicialización de Periféricos:** Llama a las APIs de Embassy para crear las instancias de los drivers (USB OTG FS y bxCAN) asignándoles sus respectivos pines (PA11/PA12 para USB; PB8/PB9 para CAN1).
- **Retorno limpio:** Agrupa estos drivers crudos en la estructura `Board` y se la entrega al `main.rs`.

### 2. `usb.rs` (Abstracción del USB)
- Vincula las interrupciones de hardware (`OTG_FS`) con el manejador de interrupciones asíncrono de Embassy usando la macro `bind_interrupts!`.
- Reserva de memoria segura para los Endpoints (mediante `StaticCell`).
- Define un *Type Alias* (`BspUsbDriver`) para ocultar los genéricos complejos del STM32 (como `peripherals::USB_OTG_FS`) al resto del programa.

### 3. `can.rs` (Abstracción del CAN)
- Vincula las cuatro interrupciones de hardware necesarias para el controlador `bxCAN` del STM32 (RX0, RX1, SCE para errores, y TX).
- Exporta el *Type Alias* `BspCan`. Deja el periférico CAN en modo de inicialización para que la lógica superior (la PC vía `gs_usb`) pueda configurar los *Bit Timings* antes de encenderlo realmente.

## 📦 Dependencias (Otros Crates que utiliza)
Para lograr su cometido, este BSP se apoya fuertemente en el ecosistema de Embassy:

1. **`embassy-stm32`**: Es el HAL (Hardware Abstraction Layer) asíncrono oficial del proyecto Embassy para chips STM32. Proporciona acceso seguro a los registros, configuración de los relojes (módulo `rcc`), y los drivers base para USB y CAN.
2. **`static_cell`**: Una micro-librería crucial en sistemas embebidos. Permite inicializar de forma segura y estática variables en memoria que vivirán por siempre (`'static`). Esto evita tener que usar código inseguro (`unsafe`) al momento de pasarle buffers de memoria a los periféricos de hardware (como los buffers USB DMA).

## 🦀 Conceptos de Rust importantes de entender aquí
Si estás leyendo o modificando este crate, hay varios patrones de Rust embebido que debes comprender:

### 1. Entorno `#![no_std]`
Al inicio de `lib.rs` notarás `#![no_std]`. Esto le indica al compilador de Rust que no debe enlazar la biblioteca estándar (`std`). En microcontroladores no tenemos un Sistema Operativo subyacente que nos proporcione memoria dinámica manejada (heap), hilos de SO, o acceso a red convencional. Solo usamos `core` (la funcionalidad más básica de Rust).

### 2. Lifetimes `'static` (Tiempos de vida)
En los *type aliases* verás `Driver<'static, ...>`. El tiempo de vida estático significa que la referencia o los datos apuntados vivirán durante toda la ejecución del programa. En sistemas embebidos, los drivers de hardware no se crean y se destruyen; se inicializan al arrancar y existen hasta que se apaga la placa. Por eso el buffer que devuelve `usb::get_ep_out_buffer()` retorna una referencia mutable estática `&'static mut [u8; 256]`.

### 3. Type Aliasing para encapsulación (`pub type`)
El tipo real de nuestro driver USB es algo así como:
```rust
embassy_stm32::usb::Driver<'static, embassy_stm32::peripherals::USB_OTG_FS>
```
Exponer esto fuera del BSP ensuciaría la aplicación principal y la acoplaría fuertemente al chip STM32. Al usar `pub type BspUsbDriver = ...`, la aplicación de arriba solo maneja `BspUsbDriver`. Si el día de mañana cambiamos a un chip RP2040, el tipo interno cambiará, pero la firma pública `BspUsbDriver` seguirá siendo la misma.

### 4. Macros para "Pegar" interrupciones (`bind_interrupts!`)
En Rust embebido, las ISR (Interrupt Service Routines) tradicionalmente requerían escribir funciones específicas en el linker con `unsafe`. Embassy usa la macro `bind_interrupts!` para enrutar de forma segura las señales eléctricas del microcontrolador (ej. llegó un mensaje CAN) hacia los "Wakers" de la máquina de estados asíncrona (`async/await`) de Rust, despertando a nuestras tareas dormidas sin bloquear el procesador.
